## 🔒 工作空间安全操作规范

### ⚠️ 强制性安全约束

**绝对禁止写入或修改任何非工作空间以外的文件和目录！**

- ✅ **允许操作**：仅能读取、修改、创建当前工作空间（`v:\4_mydoc\javascript\tauri\chchess`）中的文件和目录
- ❌ **禁止操作**：包括执行命令会引起的写入和修改任何非工作空间以外的文件和目录
- ❌ **禁止命令**：所有终端命令必须在工作空间内执行，禁止执行会导致工作空间外文件修改的命令

### 具体规则

1. 文件读写操作仅限于工作空间内的文件
2. 终端命令执行不得影响工作空间外的任何文件或目录
3. 配置文件应存储在系统配置目录（使用 `dirs` 库），但逻辑上仍属于工作空间的一部分
4. Git 操作仅针对工作空间内的代码

## 📝 Git 提交与推送规范

### Commit 规范

1. **每次修改代码前必须先执行 `git commit` 保存当前状态**
2. **Git commit message 必须严格使用单行注释**
   - ✅ 正确示例：`git commit -m "修复棋子移动规则验证bug"`
   - ❌ 错误示例：多行message、换行描述
3. **保持简洁明了**，不要冗长描述

### Push 规范

1. 这是个人项目，验证修改无误后，可自动执行 `git push` 提交到远程仓库
2. Git push 允许提交到远程仓库，前提是所有本地修改仅在工作空间内进行

### 工作流程

```bash
git add .
git commit -m "简洁的单行描述"
# 验证修改正确性
git push
```

## 💻 终端环境选择规范

### ⚠️ 禁止使用 PowerShell

**在 Windows 环境下执行 Tauri 相关命令时，禁止使用 PowerShell，会导致失败！**

### 推荐使用的终端环境

1. ✅ **CMD（命令提示符）**
2. ✅ **Git Bash**

### 🔧 Rust 编译环境配置（重要！）

**执行任何 Rust 相关的命令前，必须先执行 `vcvars64.bat` 初始化 Visual Studio 编译环境，否则 Rust 会启动失败！**

#### 操作步骤

1. **打开 CMD 或 Git Bash**
2. **执行 vcvars64.bat**（路径根据 Visual Studio 版本可能不同）：
   ```batch
   "C:\Program Files\Microsoft Visual Studio\2022\Community\VC\Auxiliary\Build\vcvars64.bat"
   ```
   或者对于 Professional 版本：
   ```batch
   "C:\Program Files\Microsoft Visual Studio\2022\Professional\VC\Auxiliary\Build\vcvars64.bat"
   ```
3. **然后执行 Rust 相关命令**：
   ```bash
   npm run tauri dev
   # 或其他 cargo 命令
   ```

#### 常见 vcvars64.bat 路径

- **Visual Studio 2022 Community**:
  `C:\Program Files\Microsoft Visual Studio\2022\Community\VC\Auxiliary\Build\vcvars64.bat`
  
- **Visual Studio 2022 Professional**:
  `C:\Program Files\Microsoft Visual Studio\2022\Professional\VC\Auxiliary\Build\vcvars64.bat`
  
- **Visual Studio 2019**:
  `C:\Program Files (x86)\Microsoft Visual Studio\2019\Community\VC\Auxiliary\Build\vcvars64.bat`

#### 为什么需要执行 vcvars64.bat？

- 设置 MSVC 编译器环境变量
- 配置正确的链接器路径
- 确保 Rust 能找到 Windows SDK 和编译工具
- 避免链接器错误和编译失败

### 特别注意

- ❌ **运行 `npm run tauri dev` 等Tauri命令时必须避免PowerShell**
- ✅ **Rust编译相关操作必须在执行 vcvars64.bat 后进行**
- PowerShell可能导致：
  - Rust链接器路径问题（使用Git Bash的`/usr/bin/link.exe`而非MSVC链接器）
  - 环境变量解析差异
  - 命令执行兼容性问题

## 🛠️ 技术栈配置

### 核心技术栈

- **前端框架**：Vue 3.5 + TypeScript 5.6
- **3D 渲染**：Three.js 0.184
- **状态管理**：Pinia 3.0

- **后端语言**：Rust
- **AI 引擎**：Pikafish (UCI协议)

### 配置要点

- Pikafish 引擎路径不写死，后期可配置

- 窗口移动/调整大小时有2秒防抖时间，避免频繁保存

## 📐 项目规范

### 中国象棋数据展示规范

1. **棋子名称必须使用传统写法**：
   - 红方用"俥、傌"
   - 黑方用"車、馬、砲"

2. **着法记录格式**：
   - 格式："颜色 + 棋子名 + 起始列 + 方向 + 距离/目标列"
   - 示例："红俥进二"

3. **UI 显示规则**：
   - 着法记录按回合分组
   - 每行显示红黑双方各一步
   - 保留 UCI 格式用于引擎通信

### FEN 格式规范

1. 棋子符号需遵循国际象棋标准缩写（如马用N，相用B），而非自定义缩写（如H,E）
2. 行棋方标识符应遵循国际象棋标准，使用'w'表示先手（红方），'b'表示后手（黑方）
3. 确保每行格子数总和为9，总行数为10，符合棋盘标准尺寸
4. 引擎通信前需验证FEN合法性，避免因格式错误导致解析失败

### 坐标系统规范

- **内部坐标**：board[0] = 黑方底线，board[9] = 红方底线
- **UCI 坐标**：row=0 = 红方底线，row=9 = 黑方底线
- **转换公式**：`uciRow = 9 - internalRow`

## 🎯 经验教训总结

### AI 引擎集成规范

1. **通信协议**：采用UCI协议进行通信，启动时运行外部exe进程，通过标准输入输出发送命令并读取响应
2. **坐标映射**：必须严格对齐内部棋盘坐标与UCI标准，转换函数需显式反转行号
3. **状态一致性**：在关键节点添加日志，打印棋盘状态、FEN串、UCI着法及转换后的坐标
4. **输出验证**：不能直接信任AI返回的着法，必须在执行移动前调用本地规则验证函数进行检查

### Rust 并发编程规范

1. **Mutex 类型设计**：应使用 `Mutex<Option<T>>` 而非 `Option<Mutex<T>>`
2. **资源管理**：若将 `Mutex<Child>` 包裹在 `Option` 中，必须先对 `Option` 进行解包获取内部引用，再调用 `lock()` 方法

### Three.js 最佳实践

1. **粗线渲染**：推荐使用`Line2`和`LineMaterial`替代标准`THREE.Line`
2. **材质安全访问**：操作材质前需进行安全检查，部分几何体使用多个材质需遍历数组
3. **对象选中反馈**：优先采用物理位移（如Y轴抬起）而非材质颜色变化

### 复杂规则验证策略

实现复杂规则验证时建议采用分层结构：
1. 第一层：基础移动逻辑（起始点、结束点、路径）
2. 第二层：交互逻辑（吃子合法性、己方棋子保护）
3. 第三层：全局状态约束（是否解将、长将限制等）

每一层验证失败时应打印详细日志以便定位问题，且需确保状态回滚正确。
